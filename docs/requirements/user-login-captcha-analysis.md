# 用户登录增加验证码校验 - 需求分析

## 一、需求核心目标和用户价值

### 核心目标
在用户连续登录失败达到指定次数（可配置，默认3次）后，强制触发验证码校验，防止恶意攻击者通过暴力破解方式猜测用户密码。

具体要求：
- 验证码为4位，包含数字和英文字母，随机生成
- 连续失败次数 ≥ 配置阈值后，登录必须输入验证码
- 每次点击登录都需要重新生成新的验证码
- 登录校验需要同时验证用户名密码 + 验证码

### 用户价值
1. **提升账户安全性**：有效阻挡暴力破解密码攻击，降低账户被盗风险
2. **防范恶意攻击**：阻止 credential stuffing（凭证填充）攻击和爆破工具
3. **保护系统资源**：减少无效登录请求对服务器资源的消耗
4. **可配置灵活性**：阈值可配置，运营可根据实际安全情况调整

## 二、已明确需求（更新后）

根据更新后的需求文档，以下问题已得到明确：

| 项 | 确认内容 |
|----|----------|
| 统计窗口 | 统计**15分钟内**的失败次数 |
| 统计维度 | 优先按 **IP地址** 统计，无法获取IP则按**用户名/邮箱**统计 |
| 验证码存储 | 存储在 **Redis**，**5分钟** 过期 |
| 验证码获取 | 后端只生成**验证码内容**（字母+数字），**前端负责生成图片** |
| 大小写敏感 | 验证码校验 **不区分大小写** |
| 验证码错误处理 | 验证码错但用户名密码正确 → **失败次数不累加**，但仍然无法登录 |
| 配置方式 | 全局统一配置 |
| IP白名单 | 支持配置信任IP白名单，白名单内IP不触发验证码 |
| 验证码规则 | 4位，必须包含数字和英文字母，随机生成 |
| 验证码刷新 | 每次点击登录都需要**重新生成**新验证码 |

## 已全部明确 ✓

所有问题都已得到澄清：

| 项 | 确认内容 |
|----|----------|
| 计数器重置 | 登录成功后，**立即清零**失败计数器 |
| 获取验证码 | 前端提前获取，需要新增 **`GetCaptcha` API接口**，返回验证码ID和验证码内容 |
| IP白名单格式 | 支持网段配置，配置格式为**数组形式** |
| 触发条件 | IP维度 **或** 用户名维度 **其中一个达到阈值就触发** |

现在需求已经完整且明确，可以开始实施。

## 三、与现有系统的交互点和潜在冲突

### 交互点识别

| 现有模块 | 交互方式 | 需要变更 |
|----------|----------|----------|
| **API接口 (auth.v1.Auth)** | Login接口需要增加验证码字段 | 需要修改proto定义，新增 `captcha` 字段；可能需要新增 `GetCaptcha` 接口 |
| **Controller (service/AuthService)** | 从请求中获取验证码参数，传递给UseCase | 需要修改 `Login` 方法参数 |
| **Business (AuthUseCase)** | 现有登录流程需要插入验证码校验步骤；需要新增：查询失败次数 → 判断是否需要验证码 → 校验验证码 → 更新失败计数 | 需要修改 `Login` 方法，新增方法查询统计连续失败次数 |
| **Domain (Repo Interface)** | 需要新增统计失败次数的接口方法；验证码存储如果需要新表也需要定义接口 | 需要扩展 `UserLoginLogRepo` 接口 |
| **DAO (UserLoginLogRepo)** | 需要实现「统计指定用户名/IP在时间窗口内失败次数」的查询 | 需要新增查询方法 |
| **Database (tbl_user_login_logs)** | 现有表结构已经存储了每次登录尝试，可以基于此统计。但分表设计可能影响查询 | 现有表结构够用，不需要改表 |
| **Redis Infrastructure** | 配置结构已定义，但data层还未初始化Redis客户端 | 需要在 `data.Data` 结构体中增加redis客户端，并完成初始化 |
| **Configuration** | 需要新增验证码开关、触发阈值、窗口时间等配置 | 需要在 `conf.Auth` 或新增 `conf.Captcha` 结构增加配置项 |
| **Captcha Storage** | 需要新增验证码存储接口和Redis实现 | 需要在domain层定义接口，data层实现 |

### 潜在冲突和风险

#### 1. **Redis客户端未初始化** ⚠️
配置结构 `conf.Data` 中已经定义了 `Redis` 配置项，但 `internal/data/data.go` 中还没有实际初始化Redis客户端，这需要在实现过程中补充。

#### 2. **分表设计对查询的影响** ✅ 已解决
现有登录日志 `tbl_user_login_logs` 按 `user_id % 10` 分表。由于我们采用Redis缓存失败计数，不需要从数据库查询统计，因此避免了跨分表查询问题。数据库仍保留原有登录日志记录用于审计即可。

#### 3. **接口兼容性风险**
修改 `LoginRequest` 增加验证码字段，该字段为可选字段，只有达到阈值才会校验，对没有传入验证码的旧客户端仍然兼容，兼容性风险可控。

#### 4. **依赖引入风险** ✅ 已规避
需求明确后端只生成验证码内容（字符串），前端生成图片，不需要引入图片处理第三方库，符合项目"禁止引入未经审批第三方依赖"规范。

#### 5. **IP白名单网段匹配**
需要支持CIDR网段匹配，Go标准库 `net` 包有 `ParseCIDR` 方法可直接使用，不需要引入第三方依赖。

## 四、初步实现思路（供参考）

1. **配置**：在配置中增加 `captcha.enabled`（开关）和 `captcha.threshold`（触发阈值，默认3）
2. **API**：在 `LoginRequest` 中增加可选的 `captcha` 字段，新增 `GetCaptcha` 接口返回验证码ID和图片数据（或仅返回验证码内容由前端生成）
3. **登录流程修改**：
   ```
   接收登录请求 → 统计近期失败次数 → 判断是否需要验证码
        ↓
   需要验证码 → 校验验证码是否正确
        ↓ 错误 → 返回错误，失败次数+1，重新生成验证码
        ↓ 正确 → 继续校验用户名密码
        ↓
   不需要验证码 → 直接校验用户名密码
        ↓
   登录失败 → 失败次数+1
   登录成功 → 清零失败次数，返回token
   ```
4. **存储设计**：
   - 连续失败次数：使用Redis缓存，15分钟过期
     - IP维度：key格式 `captcha:failcount:ip:{ipAddr}`
     - 用户名维度：key格式 `captcha:failcount:user:{loginBy}`
   - 验证码：使用Redis存储，key格式 `captcha:{captchaId}`，存储验证码内容，5分钟过期
   - 因为使用Redis缓存失败计数，不需要查询数据库统计，解决了分表查询问题

5. **配置项**：需要新增以下配置：
   ```yaml
   captcha:
     enabled: true           # 验证码功能开关
     threshold: 3            # 触发阈值（失败次数）
     window_minutes: 15      # 统计窗口（分钟）
     expire_minutes: 5       # 验证码过期（分钟）
     ip_whitelist: []        # IP白名单
   ```

6. **解决分表问题**：因为我们用Redis缓存失败计数，不需要从数据库查询统计，避免了跨分表查询问题。数据库仍保留原有登录日志记录用于审计。
